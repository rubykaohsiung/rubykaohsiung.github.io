Active Record 回撥

本指南將教您如何與 Active Record 物件。

閱讀本指南後，您將瞭解：

  • Active Record 物件的生命週期。
  • 如何建立響應物件生命週期中的事件的回撥方法。
  • 如何建立特殊的類來封裝回調的常見行為。

1 物件生命週期

在 rails 應用程式的正常執行期間，可能會建立、更新和銷燬物件。 Active Record 提供了此物件生命週期 的掛鉤，以便您可以控制您的應用程式及其資料。

回撥允許您在更改物件狀態之前或之後觸發邏輯。

2 回撥View

回撥是在物件生命週期的特定時刻被呼叫的方法。使用回撥，可以編寫在Active Record 物件被建立、儲存、更新、刪除、驗證或從資料庫載入時執行的程式碼。

2.1 回撥註冊

為了使用可用的回撥，您需要註冊它們。您可以將回調實現為普通方法，並使用巨集風格的類方法將它們註冊為回撥：

class User < ApplicationRecord
  validates :login, :email, presence: true

  before_validation :ensure_login_has_a_value

  private
    def ensure_login_has_a_value
      if login.nil?
        self.login = email unless email.blank?
      end
    end
end

巨集風格的類方法也可以接收一個塊。如果程式碼塊內的程式碼太短以至於可以放在一行中，請考慮使用這種樣式：

class User < ApplicationRecord
  validates :login, :email, presence: true

  before_create do
    self.name = login.capitalize if name.blank?
  end
end

回撥也可以註冊為僅在某些生命週期事件上觸發：

class User < ApplicationRecord
  before_validation :normalize_name, on: :create

  # :on takes an array as well
  after_validation :set_location, on: [ :create, :update ]

  private
    def normalize_name
      self.name = name.downcase.titleize
    end

    def set_location
      self.location = LocationService.query(self)
    end
end

將回調方法宣告為私有被認為是一種很好的做法。如果將它們設為 public，則可以從 Model 外部呼叫它們並違反物件封裝原則。

警告。 after_save 在建立和更新時執行，但總是after 更具體的回撥 after_create 和 after_update，無論巨集呼叫的執行順序如何。

警告。避免在回撥中更新或儲存屬性。例如，不要在回撥中呼叫 update(attribute: "value")。這可能會改變 Model 的狀態，並可能在提交期間導致意外的副作用。相反，您可以安全地在 before_create / before_update 或更早的回撥中直接分配值（例如，self.attribute = "value"）。

注意：before_destroy 回撥應該放在 dependent::destroy 之前 Associations（或使用 prepend: true 選項），以確保它們在執行之前 記錄被dependent::destroy刪除。

3.4 after_initialize 和 after_find

[after_initialize][] 回撥將在 Active Record 物件例項化時呼叫，無論是直接使用 new 還是從資料庫載入記錄。避免直接覆蓋您的 Active Record initialize 方法會很有用。

[after_find][] 回撥將在Active Record 從資料庫載入記錄時呼叫。如果兩者都已定義，則在 after_initialize 之前呼叫 after_find。

after_initialize 和 after_find 回撥沒有對應的 before_*，但它們可以像其他 Active Record 回撥一樣註冊。

class User < ApplicationRecord
  after_initialize do |user|
    puts "You have initialized an object!"
  end

  after_find do |user|
    puts "You have found an object!"
  end
end

irb> User.new
You have initialized an object!
=> #<User id: nil>

irb> User.first
You have found an object!
You have initialized an object!
=> #<User id: 1>

[after_find]：https://api.Rubyonrails.org/classes/activerecord/callbacks/classmethods.html#method-i-after_find [after_initialize]：https://api.Rubyonrails.org/classes/activerecord/callbacks/classmethods.html#method-i-after_initialize

3.5 after_touch

每當觸控Active Record 物件時，都會呼叫 [after_touch][] 回撥。

class User < ApplicationRecord
  after_touch do |user|
    puts "You have touched an object"
  end
end

irb> u = User.create(name: 'Kuldeep')
=> #<User id: 1, name: "Kuldeep", created_at: "2013-11-25 12:17:49", updated_at: "2013-11-25 12:17:49">

irb> u.touch
You have touched an object
=> true

它可以與belongs_to一起使用：

class Employee < ApplicationRecord
  belongs_to :company, touch: true
  after_touch do
    puts 'An Employee was touched'
  end
end

class Company < ApplicationRecord
  has_many :employees
  after_touch :log_when_employees_or_company_touched

  private
    def log_when_employees_or_company_touched
      puts 'Employee/Company was touched'
    end
end

irb> @employee = Employee.last
=> #<Employee id: 1, company_id: 1, created_at: "2013-11-25 17:04:22", updated_at: "2013-11-25 17:05:05">

irb> @employee.touch # triggers @employee.company.touch
An Employee was touched
Employee/Company was touched
=> true

[after_touch]：https://api.Rubyonrails.org/classes/activerecord/callbacks/classmethods.html#method-i-after_touch

4 執行回撥

以下方法觸發回撥：

*建立 *建立！ *破壞 *摧毀！ * destroy_all * destroy_by *儲存 *儲存！ *儲存（驗證：假） *切換！ *觸控 * update_attribute *更新 *更新！ *有效？

此外，after_find 回撥由以下 finder 方法觸發：

全部 * 第一 *查詢 * find_by * `find_by_ *find_by_*! *find_by_sql *最後`

每次初始化類的新物件時都會觸發 after_initialize 回撥。

注意：find_by_* 和 find_by_*! 方法是為每個屬性自動生成的動態查詢器。在動態查詢器部分 瞭解有關它們的更多資訊

5 跳過回撥

與驗證一樣，也可以使用以下方法跳過回撥：

  • 遞減！ *遞減計數器 *刪除
  • delete_all
  • delete_by
  • 增量！
  • 增量計數器
  • 插入 *插入！
  • insert_all
  • insert_all!
  • touch_all
  • update_column
  • update_columns
  • update_all
  • update_counters
  • 更新插入
  • upsert_all

但是，應謹慎使用這些方法，因為重要的業務規則和應用程式邏輯可能會保留在回撥中。在不瞭解潛在影響的情況下繞過它們可能會導致無效資料。

6 停止執行

當您開始為 Model 註冊新的回撥時，它們將排隊等待執行。該佇列將包括您所有Model 的驗證、註冊的回撥以及要執行的資料庫操作。

整個回撥鏈都包含在一個 transAction 中。如果任何回撥引發異常，則執行鏈將停止併發出回滾。故意停止鏈使用：

throw :abort

警告。回撥鏈停止後，Rails 將重新引發任何不是 ActiveRecord::Rollback 或 ActiveRecord::RecordInvalid 的異常。引發除 ActiveRecord::Rollback 或 ActiveRecord::RecordInvalid 以外的異常可能會破壞不期望像 save 和 update（通常嘗試返回 true 或 false）這樣的方法引發的程式碼一個例外。

7 關係回撥

回撥通過Model 關係工作，甚至可以由它們定義。假設一個使用者有很多文章的例子。如果使用者被銷燬，使用者的文章應該被銷燬。讓我們通過它與 article @{​​17} 的關係向 user Model 新增一個 after_destroy 回撥：

class User < ApplicationRecord
  has_many :articles, dependent: :destroy
end

class Article < ApplicationRecord
  after_destroy :log_destroy_Action

  def log_destroy_Action
    puts 'Article destroyed'
  end
end

irb> user = User.first
=> #<User id: 1>
irb> user.articles.create!
=> #<Article id: 1, user_id: 1>
irb> user.destroy
Article destroyed
=> #<User id: 1>

8 條件回撥

與驗證一樣，我們還可以根據給定謂詞的 satisfaction 呼叫回撥方法。我們可以使用 :if 和 :unless 選項來做到這一點，它們可以採用 Symbol、proc 或 array。當你想指定在什麼條件下應該呼叫回撥時，你可以使用:if選項。如果你想指定回撥 ** 不應該** 被呼叫的條件，那麼你可以使用 :unless 選項。

8.1 使用:if 和:unless 和Symbol

您可以將 :if 和 :unless 選項與 Symbol 相關聯，該 Symbol 對應於將在回撥之前被呼叫的謂詞方法的名稱。使用:if 選項時，如果謂詞方法返回false，則不會執行回撥；使用:unless 選項時，如果謂詞方法返回true，則不會執行回撥。這是最常見的選擇。使用這種註冊形式，還可以註冊幾個不同的謂詞，這些謂詞應該被呼叫以檢查是否應該執行回撥。

class Order < ApplicationRecord
  before_save :normalize_card_number, if: :paid_with_card?
end

8.2 將:if 和:unless 與Proc 一起使用

可以將 :if 和 :unless 與一個 Proc 物件相關聯。此選項最適合編寫簡短的驗證方法，通常是單行程式碼：

class Order < ApplicationRecord
  before_save :normalize_card_number,
    if: Proc.new { |order| order.paid_with_card? }
end

由於 proc 是在物件的上下文中計算的，因此也可以將其寫為：

class Order < ApplicationRecord
  before_save :normalize_card_number, if: Proc.new { paid_with_card? }
end

8.3 同時使用 :if 和 :unless

回撥可以在同一個宣告中混合使用 :if 和 :unless：

class Comment < ApplicationRecord
  before_save :filter_content,
    if: Proc.new { forum.parental_control? },
    unless: Proc.new { author.trusted? }
end

8.4 多個回撥條件

:if 和 :unless 選項也接受一組過程或方法名稱作為 Symbols：

class Comment < ApplicationRecord
  before_save :filter_content,
    if: [:subject_to_parental_control?, :untrusted_author?]
end

回撥僅在所有 :if 條件和所有 :unless 條件都被評估為 true 時執行。

9 回撥類

有時，您將編寫的回撥方法非常有用，可以被其他Model 重用。 active record 使得建立封裝回調方法的類成為可能，因此它們可以被重用。

這是一個示例，我們為 picturefile @{​​17} 建立一個帶有 after_destroy 回撥的類：

class PictureFileCallbacks
  def after_destroy(picture_file)
    if File.exist?(picture_file.filepath)
      File.delete(picture_file.filepath)
    end
  end
end

當在類中宣告時，如上所述，回撥方法將接收Model 物件作為引數。我們現在可以在Model 中使用回撥類：

class PictureFile < ApplicationRecord
  after_destroy PictureFileCallbacks.new
end

請注意，我們需要例項化一個新的 PictureFileCallbacks 物件，因為我們將回調宣告為一個例項方法。如果回撥使用例項化物件的狀態，這將特別有用。然而，通常將回調宣告為類方法會更有意義：

class PictureFileCallbacks
  def self.after_destroy(picture_file)
    if File.exist?(picture_file.filepath)
      File.delete(picture_file.filepath)
    end
  end
end

如果以這種方式宣告回撥方法，則不需要例項化 PictureFileCallbacks 物件。

class PictureFile < ApplicationRecord
  after_destroy PictureFileCallbacks
end

您可以在回撥類中宣告任意數量的回撥。

10 transAction 回撥

有兩個額外的回撥由資料庫 transaction 的完成觸發：[after_commit][] 和 [after_rollback][]。這些回撥與 after_save 回撥非常相似，除了它們在資料庫更改提交或回滾後才會執行。當您的active recordModel 需要與不屬於資料庫 transaction 的外部系統進行互動時，它們最有用。

例如，考慮前面的示例，其中picturefileModel 需要在相應記錄銷燬後刪除檔案。如果在呼叫 after_destroy 回撥並且 transaction 回滾後有任何內容引發異常，則檔案將被刪除，Model 將處於不一致的狀態。例如，假設下面程式碼中的 picture_file_2 無效並且 save! 方法引發錯誤。

picturefile.transAction do
  picture_file_1.destroy
  picture_file_2.save!
end

通過使用 after_commit 回撥，我們可以解釋這種情況。

class PictureFile < ApplicationRecord
  after_commit :delete_picture_file_from_disk, on: :destroy

  def delete_picture_file_from_disk
    if File.exist?(filepath)
      File.delete(filepath)
    end
  end
end

注意：:on 選項指定何時觸發回撥。如果你 不要提供:on 選項，回撥將為每個Action 觸發。

由於僅在建立、更新或刪除時使用 after_commit 回撥是 常見的是，這些操作有別名：

  • [after_create_commit][]
  • [after_update_commit][]
  • [after_destroy_commit][]
class PictureFile < ApplicationRecord
  after_destroy_commit :delete_picture_file_from_disk

  def delete_picture_file_from_disk
    if File.exist?(filepath)
      File.delete(filepath)
    end
  end
end

警告。當 transaction 完成時，會為在該 transaction 內建立、更新或銷燬的所有 Model 呼叫 after_commit 或 after_rollback 回撥。但是，如果在這些回撥之一中引發異常，則異常將冒泡並且任何剩餘的 after_commit 或 after_rollback 方法將不被執行。因此，如果您的回撥程式碼可能引發異常，您需要在回撥中拯救它並處理它，以允許其他回撥執行。

警告。在 after_commit 或 after_rollback 回撥中執行的程式碼本身並不包含在 transAction 中。

警告。使用具有相同方法名稱的 after_create_commit 和 after_update_commit 將只允許定義的最後一個回撥生效，因為它們都在內部別名為 after_commit，後者覆蓋先前定義的具有相同方法名稱的回撥。

class User < ApplicationRecord
  after_create_commit :log_user_saved_to_db
  after_update_commit :log_user_saved_to_db

  private
  def log_user_saved_to_db
    puts 'User was saved to database'
  end
end

irb> @user = User.create # prints nothing

irb> @user.save # updating @user
User was saved to database

還有一個別名用於將 after_commit 回撥同時用於建立和更新：

  • [after_save_commit][]
class User < ApplicationRecord
  after_save_commit :log_user_saved_to_db

  private
  def log_user_saved_to_db
    puts 'User was saved to database'
  end
end

irb> @user = User.create # creating a User
User was saved to database

irb> @user.save # updating @user
User was saved to database
